3. ОПИС ВИКОНАНОЇ РОБОТИ
3.1 Загальні відомості про шаблони (патерни) проєктування
   Патерни проєктування формалізують перевірені архітектурні рішення. Класифікація Банди чотирьох поділяє їх на три групи: 
- Породжувальні патерни вирішують питання створення об'єктів. 
- Структурні відповідають за композицію класів. 
- Поведінкові патерни визначають алгоритми та способи взаємодії між об'єктами. 
   Використання патернів в об'єктно-орієнтованому проєктуванні дозволяє розробляти гнучкі програмні системи. Такі системи легше підтримувати та масштабувати. Патерни допомагають розробникам спілкуватися спільною мовою. Замість довгого опису архітектури достатньо назвати застосований патерн. Це значно прискорює процес розробки та зменшує кількість помилок на етапі проєктування.
3.2 Опис обраного патерна
     Назва патерна: Ітератор
Тип шаблона: поведінковий
     Основне призначення: забезпечення послідовного доступу до елементів складеного об'єкта без розкриття його внутрішньої структури. Патерн дозволяє клієнтському коду працювати з різними колекціями через єдиний інтерфейс.
     Передумови застосування: необхідність доступу до вмісту об'єктів без розкриття їхнього внутрішнього представлення. Потреба у підтримці різних видів обходу для складених об'єктів. Забезпечення єдиного інтерфейсу для обходу різних структур даних. Клієнтський код не повинен залежати від конкретних класів колекцій.
     Переваги: спрощення інтерфейсу класу агрегату. Підтримка різних алгоритмів обходу. Можливість одночасного використання кількох ітераторів для однієї структури. Відповідність принципу єдиної відповідальності. Логіка обходу виноситься в окремий клас. Це дозволяє змінювати алгоритми обходу без модифікації самої колекції.
     Недоліки: невиправдане ускладнення коду у разі використання для простих масивів або базових колекцій. Збільшення загальної кількості класів у системі. Потенційне зниження продуктивності під час обходу порівняно з прямим доступом до елементів за індексом. Кожен запит до наступного елемента вимагає виклику методу, що створює додаткове навантаження.
3.3 Структура шаблона
   Опис структурних частин:
- Ітератор визначає базовий інтерфейс для доступу та обходу елементів. Він містить методи для отримання поточного елемента та переходу до наступного.
- Конкретний ітератор реалізує цей інтерфейс і відстежує поточну позицію під час виконання обходу. Він безпосередньо взаємодіє з колекцією для отримання даних.
- Агрегат визначає інтерфейс для створення об'єкта-ітератора. Він гарантує, що кожна колекція зможе надати відповідний ітератор.
- Конкретний агрегат: реалізує цей метод і повертає екземпляр відповідного конкретного ітератора, пов'язаного з поточною колекцією.
   Опис взаємодії об'єктів:
   Клієнтський код надсилає запит на створення ітератора через фабричний метод об'єкта агрегату. Агрегат створює конкретний ітератор та ініціалізує його посиланням на власні дані. Після цього агрегат повністю делегує відповідальність за обхід створеному ітератору. Клієнт послідовно запитує наступні елементи саме через екземпляр ітератора. Перед кожним запитом клієнт перевіряє стан завершення обходу. Це виключає вихід за межі колекції.
   UML-діаграма з поясненнями:

Рисунок 3.1 – UML-діаграма класів патерну Iterator 
     На UML-діаграмі (див. рис 3.1) відображено незалежність клієнта від конкретних реалізацій колекцій та ітераторів. Клієнт взаємодіє виключно з базовими абстрактними інтерфейсами. Конкретний агрегат пов'язаний з Конкретним ітератором відношенням створення. Він відповідає за інстанціювання правильного ітератора. Конкретний ітератор має зв'язок з Конкретним агрегатом для доступу до його внутрішніх елементів. Залежності між клієнтом та підсистемою колекцій побудовані через абстракції, що забезпечує гнучкість архітектури.
3.4 Реалізація шаблона
     Опис реалізації мовою програмування Python.
     Основні етапи:
* Створення базових інтерфейсів. На цьому етапі проєктується абстракція для ітератора та колекції. Розробка базового абстрактного класу ітератора вимагає визначення методів отримання наступного елемента та перевірки наявності доступних елементів. Створення базового абстрактного класу колекції включає оголошення фабричного методу для створення ітератора.
* Реалізація конкретних класів. На цьому етапі створюються класи, які містять реальну логіку. Конкретна колекція реалізує внутрішнє зберігання даних у вигляді списку. Вона також надає методи для додавання нових елементів та отримання їх кількості. Її відповідний ітератор реалізує логіку послідовного переміщення по цих даних за допомогою внутрішнього індексу.
* Демонстрація використання. Цей етап показує інтеграцію створених класів у клієнтському коді. Відбувається ініціалізація об'єкта конкретної колекції та наповнення її тестовими даними. Далі виконується виклик методу колекції для отримання готового до роботи екземпляра ітератора. Наприкінці відбувається використання отриманого об'єкта в циклі для послідовного виведення всіх елементів на екран.3.5 Організація GitHub-репозиторію

1 from abc import ABC, abstractmethod
2 from typing import List, Optional

3 # --- Абстракції патерна ---

4 class Iterator(ABC):
5 """
6 ПАТЕРН: Абстрактний Ітератор
7 Визначає інтерфейс для обходу елементів колекції.
8 """
9 @abstractmethod
10 def has_next(self) -> bool:
"""Перевіряє, чи є ще елементи для обходу."""
pass

11 @abstractmethod
12 def next(self):
"""Повертає наступний елемент колекції."""
pass


13 class Aggregate(ABC):
14 """
15 ПАТЕРН: Абстрактний Агрегат
16 Визначає інтерфейс для створення об'єкта-ітератора.
17 """
18 @abstractmethod
19 def create_iterator(self) -> Iterator:
pass


20 # --- Сутності предметної області ---

21 class Song:
22 """
23 Клас, що представляє окремий елемент колекції (пісню).
24 """
25 def __init__(self, title: str, artist: str):
self.title = title
self.artist = artist

26 def __str__(self) -> str:
return f"{self.artist} - {self.title}"


27 # --- Конкретні реалізації патерна ---

28 class PlaylistIterator(Iterator):
29 """
30 ПАТЕРН: Конкретний Ітератор
31 Реалізує алгоритм обходу конкретної колекції (плейлиста).
32 Зберігає поточний стан обходу.
33 """
34 def __init__(self, playlist: 'Playlist'):
# Зберігаємо посилання на агрегат, який будемо обходити
self._playlist = playlist
# Початковий стан ітератора
self._current_index = 0

35 def has_next(self) -> bool:
# Перевіряємо, чи індекс не вийшов за межі списку пісень
return self._current_index < len(self._playlist.get_songs())

36 def next(self) -> Optional[Song]:
# Повертаємо поточну пісню і зміщуємо індекс на один крок вперед
if self.has_next():
song = self._playlist.get_songs()[self._current_index]
self._current_index += 1
return song
return None


37 class Playlist(Aggregate):
38 """
39 ПАТЕРН: Конкретний Агрегат
40 Зберігає колекцію об'єктів та повертає відповідний ітератор.
41 """
42 def __init__(self):
self._songs: List[Song] = []

43 def add_song(self, song: Song) -> None:
self._songs.append(song)

44 def get_songs(self) -> List[Song]:
"""Допоміжний метод для доступу до списку з ітератора."""
return self._songs

45 def create_iterator(self) -> Iterator:
"""
Реалізація фабричного методу для створення ітератора,
пов'язаного з цим конкретним плейлистом.
"""
return PlaylistIterator(self)
