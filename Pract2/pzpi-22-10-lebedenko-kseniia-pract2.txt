2. ЗАВДАННЯ

   Метою роботи є дослідження архітектури складної програмної системи та аналіз застосованих архітектурних підходів.

   У межах роботи необхідно:
- виконати аналіз архітектури програмної системи;
- визначити архітектурний стиль;
- описати основні компоненти;
- проаналізувати взаємодію підсистем;
- оцінити масштабованість та переваги архітектурних рішень;
- підготувати презентацію та відеодоповідь;
- оформити звіт відповідно до ДСТУ 3008:2015.

3. ОПИС ВИКОНАНОЇ РОБОТИ
3.1 Загальні відомості про систему
     Назва системи: Docker
     Розробник: 	Solomon Hykes; Docker, Inc
     Призначення системи: 
     Технологія застосовується в галузі хмарних обчислень, забезпечення процесів безперервної інтеграції та для організації локальних середовищ програмування. Основними користувачами є розробники програмного забезпечення, інженери інфраструктури та системні адміністратори.
     Базовий функціонал полягає у пакуванні коду з усіма залежностями в ізольовані контейнери для гарантовано однакового запуску додатків у будь-яких обчислювальних середовищах.
3.2 Архітектурний стиль системи
     Архітектурний стиль системи: клієнт-серверна архітектура
     Обґрунтування вибору архітектури:
- Обрана модель дозволяє чітко відокремити користувацький інтерфейс від внутрішніх процесів управління ядром операційної системи.
- Клієнтський компонент приймає текстові команди та відправляє їх серверу через програмний інтерфейс.
- Серверний компонент бере на себе всю роботу зі створення середовищ, розподілу пам'яті та налаштування мережі.
- Розподіл ролей дає можливість клієнту та серверу працювати на різних фізичних машинах.
- Користувач отримує можливість керувати віддаленими серверами зі свого локального пристрою.
- Сервер виступає централізованою точкою контролю ресурсів і забезпечує надійну ізоляцію процесів.
- Взаємодія компонентів через стандартизований протокол зв'язку дозволяє легко інтегрувати платформу зі сторонніми інструментами розгортання.
- Тривалі завдання зі збирання образів виконуються сервером асинхронно, залишаючи термінал клієнта вільним для подальшої роботи.



3.3 Основні компоненти системи
     Інтерфейс користувача:
     Користувацький інтерфейс представлений інструментом командного рядка Docker CLI та графічним додатком Docker Desktop. Клієнт взаємодіє з сервером через програмний інтерфейс, передаючи команди на створення або керування об'єктами.
     Серверна частина:
     Серверна частина реалізована у вигляді постійного фонового процесу Docker Daemon. Сервер прослуховує запити від клієнта та виконує операції з управління життєвим циклом контейнерів і образів.
     Рівень бізнес-логіки:
     Рівень бізнес-логіки забезпечують низькорівневі програмні модулі containerd та runc. Перший компонент керує виконанням процесів на високому рівні. Другий компонент безпосередньо взаємодіє з ядром операційної системи для створення ізольованих середовищ.
     Рівень зберігання даних:
     Рівень зберігання даних включає локальну файлову систему для збереження завантажених образів. Для довготривалого збереження інформації застосовується механізм ізольованих томів. Глобальне розповсюдження файлів забезпечується зовнішніми реєстрами, основним з яких є Docker Hub.
     Допоміжні сервіси:
     Допоміжні сервіси представлені утилітою Docker Compose для запуску багатоконтейнерних середовищ. Вбудовані мережеві драйвери забезпечують безпечний зв'язок між різними компонентами системи та зовнішнім світом.
3.4 Взаємодія компонентів
     Опис потоків даних в системі:
     Потоки даних у системі ініціюються користувачем через введення команд у терміналі. Інструмент командного рядка перетворює введені текстові інструкції на структуровані запити. Згенеровані запити направляються до центрального компонента управління системою. Серверна частина приймає дані та здійснює їх валідацію перед подальшою обробкою. 
     Наступним етапом є взаємодія з підсистемою зберігання для перевірки наявності необхідних базових файлів. За відсутності локальної копії ініціюється мережевий потік завантаження з віддаленого сховища. Після забезпечення наявності всіх необхідних ресурсів формується потік інструкцій до середовища виконання. Результати роботи ізольованого процесу повертаються у зворотному напрямку для відображення користувачу.

Рисунок 3.1 – Діаграма послідовностей потоків даних
     Опис діаграми:
     На поданій діаграмі (див. рис. 3.1) послідовностей потоків даних відображено процес обробки команди запуску контейнера. 
     Взаємодія починається з передачі запиту від користувача до клієнтського додатка. Клієнтський додаток транслює запит до фонового процесу управління. Фоновий процес перевіряє наявність потрібного образу в локальному сховищі. Блок умов демонструє два можливі сценарії розвитку подій. 
     Перший сценарій передбачає відсутність образу, що викликає відправлення запиту до зовнішнього реєстру, отримання даних та їх збереження на диску. Другий сценарій спрацьовує при знаходженні локальної копії образу. Після завершення перевірки фоновий процес передає інструкції запуску до середовища виконання. Середовище виконання повертає статус операції, який асинхронно передається через клієнтський додаток для фінального відображення результату.
     Опис протоколів взаємодії:
     Архітектура системи побудована на використанні кількох рівнів мережевих та локальних протоколів. Головним стандартом спілкування між клієнтськими інструментами та серверною частиною виступає архітектурний стиль передачі стану представлення. Дані передаються у форматі стандартизованих текстових об'єктів. 
     Для комунікації в межах однієї фізичної машини застосовуються локальні системні сокети. Віддалене керування реалізується через надійні протоколи управління передачею даних. Завантаження образів із зовнішніх джерел здійснюється за допомогою захищених протоколів передачі гіпертексту. Внутрішня взаємодія серверного демона з низькорівневим середовищем виконання відбувається через високопродуктивний фреймворк віддаленого виклику процедур. Фінальний запуск процесів вимагає прямих системних викликів до ядра операційної системи.

Рисунок 3.2 – Діаграма розгортання протоколів взаємодії
     Опис діаграми розгортання:
     Діаграма розгортання (див. рис. 3.2) ілюструє фізичний розподіл компонентів та застосовані протоколи зв'язку. Клієнтський вузол містить базовий інструмент командного рядка та утиліту для запуску багатоконтейнерних додатків. Обидва інструменти використовують програмний інтерфейс передачі стану представлення через локальні сокети або мережеві з'єднання для зв'язку з вузлом хоста. Вузол хоста включає головний фоновий процес управління та два компоненти середовища виконання. Головний процес звертається до зовнішнього вузла реєстру через захищене з'єднання протоколу передачі гіпертексту. Зв'язок головного процесу з високорівневим середовищем виконання забезпечується віддаленим викликом процедур через локальний сокет. Високорівневе середовище передає команди низькорівневому компоненту безпосередньо через системні виклики ядра.
     Опис механізмів синхронізації:
     Синхронізація стану між серверною частиною та користувацьким інтерфейсом досягається за рахунок механізму підписки на системні події. Клієнт не надсилає постійні запити для перевірки статусу, а встановлює єдине довготривале з'єднання. Серверний компонент безперервно моніторить зміни у життєвому циклі підпорядкованих процесів. Кожна зміна стану фіксується у внутрішньому журналі та миттєво транслюється всім підключеним слухачам.
     Такий підхід гарантує мінімальну затримку при відображенні інформації. Механізм асинхронної передачі дозволяє клієнтській програмі оперативно оновлювати інтерфейс без блокування основного потоку виконання команд. З'єднання залишається активним до моменту явної відміни операції або повного завершення роботи цільового процесу.

Рисунок 3.3 – Діаграма станів/послідовностей механізму синхронізації
     Опис діаграми синхронізації:
     На графіку (див. рис. 3.3) показано процес встановлення та підтримки синхронізованого зв'язку між клієнтом і сервером. Взаємодія стартує з відправлення клієнтом запиту на підключення до потоку подій. Сервер відповідає відкриттям довготривалого з'єднання. 
     Наступні кроки демонструють цикл обробки внутрішніх змін. Контейнер генерує подію зміни статусу, яку фіксує серверний компонент. Зареєстрована подія одразу пересилається клієнту через механізм асинхронної передачі даних. Отримавши інформацію, клієнтський додаток здійснює оновлення локального відображення статусу.
     Аналогічний ланцюжок дій повторюється при завершенні роботи ізольованого середовища. Після фінального оновлення статусу відбувається планове закриття з'єднання між клієнтом та сервером.
3.5 Масштабованість та надійність
     Механізми балансування навантаження реалізуються через вбудовані інструменти оркестрації:
- Кластерний режим розподіляє вхідний мережевий трафік між активними вузлами системи. 
- Внутрішній мережевий маршрутизатор автоматично призначає запити на доступні екземпляри додатків.
     Відмовостійкість забезпечується постійним моніторингом стану запущених процесів. Система автоматично перезапускає компоненти у разі їх непередбаченого завершення. Розподіл завдань між різними фізичними серверами мінімізує ризики повної зупинки роботи. 
     Кешування застосовується на етапі збирання образів для прискорення розгортання. Кожен крок інструкції зберігається як окремий шар у локальній файловій системі. Наступні процеси збирання використовують готові шари замість повторного виконання ідентичних операцій.
     Реплікація даних та процесів налаштовується декларативним способом. Користувач задає бажану кількість копій додатку в конфігураційному файлі. Система самостійно створює необхідну кількість клонів та підтримує їх активний стан. 
     Контейнеризація виступає базовим елементом для побудови хмарної інфраструктури. Ізольовані середовища легко переносяться між локальними серверами та публічними хмарними платформами. Стандартизований формат пакування гарантує однакову роботу додатків незалежно від апаратного забезпечення.
3.6 Приклади програмної реалізації
     У процесі підготовки доповіді було використано інструмент штучного інтелекту Gemini 3.1 Pro. Метою використання було отримання релевантних прикладів програмного коду, опису потоків даних та детального пояснення гібридних архітектурних підходів.
     Перший фрагмент коду (див. дод. В.1) демонструє взаємодію клієнтського скрипта мовою Python з інфраструктурою хоста через інтегровану бібліотеку Docker SDK. Цей код ілюструє архітектурний принцип клієнт-серверної моделі та шаблон Фасад. Замість того, щоб скрипт безпосередньо звертався до ядра операційної системи для створення ізольованих середовищ, він взаємодіє з уже запущеним фоновим процесом управління. Виклик функції завантаження образу показує, як система абстрагує складні механізми роботи з мережею та файловою системою. Локальний клієнт передає лише коротку інструкцію без необхідності самостійно розпаковувати архіви.
     Такий підхід суттєво розвантажує клієнтські інструменти, переносячи частину обчислювального навантаження на центральний демон, та гарантує високий рівень безпеки виконання процесів.
     Другий фрагмент коду (див. дод. В.2) ілюструє внутрішню комунікацію на рівні віртуальних мереж, зокрема взаємодію веб-сервера з ізольованою базою даних. Використання декларативного конфігураційного файлу з передачею параметрів середовища демонструє принцип слабкого зв'язування та ізоляції. Веб-сервер не містить жорстко закодованих мережевих адрес. Він лише делегує завдання маршрутизації вбудованому серверу доменних імен платформи контейнеризації. Завдяки такому розподілу відповідальностей ці компоненти можна масштабувати незалежно один від одного. Якщо контейнер бази даних тимчасово вийде з ладу, веб-сервер не зупинить свою роботу фатально, а просто використає механізм автоматичного перезапуску або резервний сценарій обробки помилок.
     Третій фрагмент коду (див. дод. В.3) розкриває принципи зберігання, кешування та дистрибуції даних у системі з шаруватою архітектурою. В архітектурі Docker чітко розділені рівні доступу до файлової системи залежно від характеру даних. Для тимчасових логів та процесів виконання використовуються ефемерні шари, що гарантують чистоту середовища після перезапуску. Водночас для постійного зберігання даних бази застосовується механізм іменованих томів. Це дозволяє уникнути втрати критичної інформації при видаленні контейнера. Найважливішою архітектурною особливістю є делегування доставки важкого контенту базових образів на периферійні сервери зовнішніх реєстрів. Клієнт отримує маніфест з центрального хабу, але завантажує лише відсутні шари даних, що є класичним прикладом оптимізації мережевого трафіку та прискорення розгортання.
